# WSR 闪卡产品需求文档

最后更新：2026-06-22

## 1. 产品概述

WSR 闪卡是一个运行在 Obsidian 内的中文闪卡学习插件，面向需要在本地 Markdown 知识库中沉淀、学习和复习知识点的用户。产品以 Markdown 源文件作为卡片内容的权威来源，以插件数据保存设置、题库缓存、FSRS 学习状态和学习历史，从而在不破坏用户笔记资产的前提下提供间隔重复、刷题、词表浏览和学习统计能力。

当前版本的核心价值是：

- 让用户直接在 Obsidian Markdown 文件中维护题库，不需要导入到外部平台。
- 按配置标签自动扫描题库，每个源文件对应一个题库。
- 使用 FSRS 调度学习和复习节奏，支持新卡、到期复习和会话内重来。
- 同一份卡片内容支持学习模式、刷题模式和单词列表三种使用方式。
- 支持正常出题和反转出题，且两种方向共享同一份学习进度。
- 支持在插件界面内新增、编辑、删除题目，并写回源 Markdown 文件。
- 支持桌面端和移动端 Obsidian 使用，界面默认中文，也提供英文界面。

## 2. 目标用户

### 2.1 核心用户

- 使用 Obsidian 管理学习资料、词汇、概念或问答知识点的学习者。
- 希望题库内容长期保存在 Markdown 文件中，而不是被锁在插件私有数据库里的用户。
- 需要通过间隔重复保持记忆的用户，例如语言学习、考试复习、专业术语背诵、课程知识点复盘。

### 2.2 次要用户

- 已经维护大量 Obsidian 笔记，希望用标签将部分文件转成可学习题库的用户。
- 需要在移动端随时刷题、复习或查看词表的用户。
- 需要针对不同题库设置不同学习节奏的用户。

## 3. 核心场景

### 3.1 从 Markdown 文件生成题库

用户在 Obsidian 中创建 Markdown 文件，文件包含一个用于识别题库的首个标签，并按当前卡片格式编写题目。插件同步后将符合配置标签的文件展示为题库。

### 3.2 按每日节奏学习新卡和复习旧卡

用户进入某个题库，查看今日新卡数、待复习数、学习进度和学习日程。开始学习后，插件按 FSRS 状态和每日限制组成本轮学习队列，用户通过显示答案和评分推进卡片。

### 3.3 快速刷题验证掌握情况

用户选择题库和题量后进入刷题模式。刷题模式随机抽题，不更新 FSRS 调度，只记录本轮正确率、错题列表和用时，适合自测。

### 3.4 浏览和遮盖词表

用户进入单词列表后，可按源文件顺序或乱序浏览两列内容，也可以遮盖第一列或第二列，点击单元格临时显示内容，双击查看解释内容。

### 3.5 维护题目内容

用户可以在首页新增题目，也可以在学习或刷题过程中编辑或删除当前题目。所有内容变更必须写回源 Markdown 文件，并在写回后重新解析题库。

### 3.6 查看近期学习投入

用户在统计页查看最近 20 个有记录日期内的学习、刷题、列表浏览记录，了解学习天数、总时长和卡片数量。

## 4. 术语定义

| 术语 | 定义 |
| --- | --- |
| 卡片 | 一条可学习内容，包含正面、背面和可选解释。 |
| 题库 | 一个符合配置标签和卡片格式的 Markdown 文件。题库 ID 使用源文件路径。 |
| 标签 | Markdown 文件中的首个 hashtag，用于判断文件所属题库类型。 |
| 正面 | 卡片的 front 内容，正常模式下作为问题。 |
| 背面 | 卡片的 back 内容，正常模式下作为答案。 |
| 解释 | 卡片的 explanation 内容，可选，始终跟随答案侧展示。 |
| 正常模式 | 使用正面出题、背面作答。 |
| 反转模式 | 使用背面出题、正面作答。解释仍跟随答案展示。 |
| 学习模式 | 使用 FSRS 更新卡片记忆状态的正式学习流程。 |
| 刷题模式 | 不更新 FSRS，仅记录正确、错误、正确率和错题的自测流程。 |
| DataStore | 插件的数据边界，负责设置、题库缓存、学习历史、FSRS 状态和源文件写回。 |

## 5. 信息架构

### 5.1 Obsidian 插件入口

- 左侧 ribbon 图标：打开闪卡学习视图。
- 命令面板：
  - 打开闪卡学习。
  - 同步闪卡题库。
- 设置页：配置标签、语言、默认学习参数和 FSRS 参数。
- 自定义视图：承载 React 闪卡应用。

### 5.2 应用内页面

- 首页：题库列表、题库统计、刷新、统计入口、新增题目入口。
- 学习准备页：题库学习概览、今日任务、学习日程、学习顺序、出题方向。
- 学习页：问题、答案、解释、评分按钮、上一题、计时、进度、编辑和删除。
- 刷题准备页：题量选择、覆盖率、出题方向、规则说明。
- 刷题页：问题、答案、解释、正确/错误按钮、上一题、计时、进度、编辑和删除。
- 刷题总结页：正确率、总题数、正确数、错误数、用时、错题列表、错题重练。
- 单词列表页：两列词表、乱序、遮盖列、解释弹窗、虚拟滚动。
- 学习统计页：最近 20 天学习记录聚合。

## 6. 主要用户流程

### 6.1 题库创建和同步流程

1. 用户创建 Markdown 文件，并在文件中写入一个标签，例如 `#wordTag`。
2. 用户按当前卡片格式编写题目。
3. 用户打开闪卡视图或点击刷新。
4. 插件扫描所有 Markdown 文件。
5. 插件提取每个文件的首个标签。
6. 当标签匹配配置项且文件包含合法卡片语法时，插件将该文件解析为题库。
7. 首页展示题库名称、标签、总卡片数、新卡数、待复习数和学习次数。

### 6.2 学习模式流程

1. 用户从首页点击某题库的“学习”。
2. 系统进入学习准备页，展示今日新卡、待复习、已完成天数和学习计划。
3. 用户选择顺序学习或乱序学习。
4. 用户选择正常模式或反转模式。
5. 用户开始学习。
6. 系统按每日新卡上限和每日复习上限生成队列。
7. 用户按空格或按钮显示答案。
8. 用户按评分按钮或数字快捷键评分。
9. 评分 1-4 更新 FSRS 状态；评分 5 进入自定义“辣鸡”路径，安排 21 天后复习。
10. 如果评分为重来，本卡进入本轮 repeatQueue，会在本轮后续再次出现。
11. 本轮完成后记录题库学习次数、最后学习时间和学习历史。

### 6.3 刷题模式流程

1. 用户从首页点击某题库的“刷题”。
2. 系统进入刷题准备页，展示题库规模和可抽取题目数。
3. 用户选择快捷题量或输入自定义题量。
4. 用户选择正常模式或反转模式。
5. 系统从题库中随机抽取指定数量的卡片。
6. 用户逐题显示答案，并标记正确或错误。
7. 用户可以回到上一题，上一题答案记录会被撤销。
8. 本轮结束后进入总结页，展示正确率、错题列表和用时。
9. 用户可以重新刷题、只练错题或返回首页。

### 6.4 单词列表流程

1. 用户从首页点击某题库的“列表”。
2. 系统按源文件顺序展示卡片正面和背面两列。
3. 用户可以切换乱序或恢复顺序。
4. 用户可以遮盖第一列或第二列。
5. 遮盖状态下，用户点击单元格显示或隐藏该项内容。
6. 用户双击第一列或第二列，系统弹窗展示解释内容。
7. 用户停留达到记录阈值后返回首页，系统记录一次列表浏览历史。

### 6.5 内容维护流程

1. 用户从首页新增题目，或从学习/刷题页编辑当前题目。
2. 系统打开题目编辑弹窗。
3. 用户输入正面、背面和可选解释。
4. 系统校验正面和背面不能为空。
5. 系统禁止用户在内容中单独一行使用 `??`、`::`、`;;`，避免破坏卡片结构。
6. 保存后系统写回源 Markdown 文件。
7. 系统重新解析源文件，刷新题库缓存，并保留可保留的 FSRS 状态。

## 7. 功能需求

### FR-001 题库扫描

- 系统必须扫描 Obsidian vault 中的 Markdown 文件。
- 系统必须提取文件中的首个 hashtag 作为题库标签，标签需支持中文字符。
- 系统只应把配置标签命中的文件作为题库。
- 系统只应把同时包含 `??` 和 `;;` 结构标记的文件视为闪卡文件。
- 系统必须在同步时缓存已发现的闪卡标签，供设置页快速添加。
- 系统在解析某个文件失败时应跳过该文件并继续扫描其他文件。

### FR-002 当前卡片格式

卡片格式必须以当前实现为准：

```markdown
#wordTag

## science

??
n. 科学；自然科学；理科
::
/ˈsaɪəns/
root: sci- = know 知道
;;

## apple

??
n. 苹果
;;
```

- `??` 必须独占一行，用于分隔正面和背面。
- `::` 必须独占一行，用于分隔背面和解释。
- `::` 和解释内容是可选项。
- `;;` 必须独占一行，用于结束单张卡片。
- 正面和背面不能为空。
- 正面、背面和解释都应支持 Markdown 渲染。
- 旧格式不作为当前需求，系统不需要自动转换旧格式文件。

### FR-003 首页题库管理

- 首页必须展示题库总数、总新卡数、总待复习数、总卡片数和总学习次数。
- 每个题库卡片必须展示题库名、标签、总卡片数、新卡数、待复习数和学习次数。
- 每个题库必须提供学习、刷题、单词列表、打开源文件和题库学习设置入口。
- 首页必须提供刷新题库入口。
- 首页必须提供学习统计入口。
- 首页必须提供新增题目入口。
- 没有题库时必须展示空状态和扫描提示。

### FR-004 题库级学习设置

- 系统必须支持全局默认学习设置。
- 系统必须支持按题库覆盖每日新卡数量、每日复习数量、学习顺序和 FSRS 参数。
- 当题库未启用自定义设置时，必须使用全局默认设置。
- 题库设置弹窗必须展示当前题库卡片总数和预计完成天数。
- 用户可以通过每日新卡数量或预计完成天数调整学习节奏。
- 用户关闭自定义设置时，系统必须删除该题库的覆盖配置。

### FR-005 学习准备页

- 学习准备页必须展示题库名称、标签、总卡片数、总天数、当前阶段和今日任务数。
- 系统必须按每日新卡数量将题库拆分为学习日程。
- 已完成学习日必须显示完成状态。
- 第一个未完成学习日必须显示为当前学习日。
- 当前学习日之后的未完成日必须显示为锁定状态。
- 已完成学习日必须允许用户进行复习练习。
- 用户必须能选择顺序学习或乱序学习。
- 用户必须能选择正常模式或反转模式。

### FR-006 学习模式

- 学习模式必须按新卡和到期卡生成学习队列。
- 新卡数量不得超过有效设置中的每日新卡数量。
- 到期复习卡数量不得超过有效设置中的每日复习数量。
- 乱序学习时，学习队列必须随机化。
- 正常模式下必须用正面出题、背面作答。
- 反转模式下必须用背面出题、正面作答。
- 解释必须只在答案区域展示，且不受出题方向影响。
- 用户显示答案前不得看到答案和解释。
- 用户评分后必须更新该卡片的 FSRS 状态并持久化。
- 用户点击上一题时，应能返回上一张已答卡片。
- 用户退出学习时必须出现确认，已完成的学习记录应保留。

### FR-007 学习评分

- 学习评分必须提供 5 个按钮：重来、困难、良好、简单、辣鸡。
- 评分 1 必须映射 FSRS Again，并将卡片加入本轮重复队列。
- 评分 2 必须映射 FSRS Hard。
- 评分 3 必须映射 FSRS Good。
- 评分 4 必须映射 FSRS Easy。
- 评分 5 必须设置为 21 天后复习，并标记为复习状态。
- 评分按钮必须展示快捷键和下次间隔提示。
- 空格在显示答案后必须等价于选择“良好”。

### FR-008 刷题准备页

- 刷题准备页必须展示题库规模、当前题量和覆盖率。
- 系统必须提供 20、50、100、150、200 和全部的快捷题量。
- 快捷题量不得超过题库卡片总数。
- 用户必须能输入自定义题量。
- 自定义题量必须限制在 1 到题库总卡片数之间。
- 用户必须能选择正常模式或反转模式。
- 刷题开始后必须随机抽取指定数量卡片。

### FR-009 刷题模式

- 刷题模式必须展示当前进度、计时和出题方向。
- 用户必须先显示答案，再标记正确或错误。
- 正常模式和反转模式的展示规则必须与学习模式一致。
- 刷题模式不得更新 FSRS 调度状态。
- 用户标记正确或错误后，系统必须记录本题答案结果。
- 用户点击上一题时，必须回到上一题并撤销上一题结果。
- 刷题完成后必须生成结果对象，包含总题数、正确数、错误数、正确率、错题 ID 列表、用时和出题方向。

### FR-010 刷题总结和错题复盘

- 刷题总结页必须展示本轮完成文案、题库名、总题数和用时。
- 总结页必须展示正确率、总题数、正确数、错误数和用时。
- 当存在错题时，必须展示错题列表。
- 错题列表必须按本轮出题方向展示问题和答案。
- 错题存在解释时，必须展示解释。
- 用户必须能重新开始刷题。
- 用户必须能只练本轮错题。
- 用户必须能返回题库首页。

### FR-011 单词列表

- 单词列表必须展示题库名称、卡片数量和标签。
- 列表必须展示两列：正面和背面。
- 解释内容不得作为第三列常驻展示。
- 用户必须能切换乱序和恢复源文件顺序。
- 用户必须能遮盖或取消遮盖第一列。
- 用户必须能遮盖或取消遮盖第二列。
- 遮盖列中的单元格必须支持点击显示或再次点击隐藏。
- 用户双击第一列或第二列时，必须打开解释弹窗。
- 解释为空时，弹窗必须展示空内容提示。
- 列表必须使用虚拟滚动支持大量卡片，避免一次性渲染所有行。

### FR-012 题目新增、编辑和删除

- 首页必须提供新增题目的浮动按钮。
- 新增题目时，用户必须能选择目标题库。
- 学习和刷题过程中必须能编辑当前题目。
- 学习和刷题过程中必须能删除当前题目。
- 编辑弹窗必须包含正面、背面和解释三个输入区。
- 正面和背面必须必填。
- 解释必须可选。
- 保存时必须禁止保留结构标记独占行。
- 新增题目必须追加到目标题库源文件末尾。
- 编辑题目必须替换对应源文件卡片块。
- 删除题目必须从源文件删除对应卡片块。
- 删除题目后，当前学习或刷题队列必须移除该卡，并对后续卡片 ID 做必要重映射。
- 当删除导致当前队列为空时，系统必须回到首页。

### FR-013 Markdown 渲染

- 学习、刷题、错题总结中的卡片内容必须使用 Obsidian MarkdownRenderer 渲染。
- 系统不得用原始 HTML 注入替代 Obsidian Markdown 渲染路径。
- 渲染容器在内容更新时必须清空旧内容再渲染新内容。

### FR-014 学习统计

- 系统必须记录学习模式、刷题模式和单词列表浏览的历史。
- 历史记录必须包含日期、题库 ID、题库名、模式、卡片数量、时长和时间戳。
- 单词列表浏览必须达到最小时长后才记录。
- 系统只保留最近 20 个有学习记录日期的数据。
- 统计页必须按日期倒序展示。
- 统计页必须展示学习天数、总时长和总卡片数。
- 每日记录必须展示当天记录数、总时长和卡片数。
- 每条记录必须展示模式、题库名、卡片数和时长。

### FR-015 设置页

- 设置页必须支持配置多个闪卡标签。
- 设置页必须支持刷新已发现的闪卡标签。
- 设置页必须支持刷新并清理无效配置标签。
- 设置页必须支持点击已发现标签快速添加到配置。
- 设置页必须支持切换中文和英文界面。
- 设置页必须支持配置每日新卡数量，范围为 1 到 200。
- 设置页必须支持配置每日复习数量，范围为 1 到 500。
- 设置页必须支持配置默认学习顺序。
- 设置页必须支持配置目标记忆保持率，范围为 0.70 到 0.99。
- 设置页必须支持配置最大复习间隔，范围为 30 到 3650 天。
- 设置页必须展示当前卡片格式和快捷键说明。
- 设置页必须保留声明式设置不可用时的手动渲染兜底能力，避免设置页空白。

### FR-016 双语界面

- 系统默认语言必须为中文。
- 系统必须支持英文界面。
- 切换语言后，ribbon、命令名、主视图标题和 React 界面文案必须更新。
- 练习完成文案应根据语言使用对应默认值。
- 用户自定义过练习文案时，系统必须保留用户自定义内容。

### FR-017 快捷键

- 学习和刷题页面必须忽略输入框、文本域中的全局快捷键。
- 空格必须在未显示答案时显示答案。
- 学习模式中，答案显示后：
  - `1` 选择重来。
  - `2` 选择困难。
  - `3` 或空格选择良好。
  - `4` 选择简单。
  - `5` 选择辣鸡。
  - `6` 返回上一题。
- 刷题模式中，答案显示后：
  - `1` 或 `X` 标记错误。
  - `2`、`O` 或空格标记正确。
  - `6` 返回上一题。

## 8. 数据与持久化

### 8.1 数据权威边界

- Markdown 源文件是卡片内容的权威来源。
- 插件数据是设置、题库缓存、FSRS 状态和学习历史的权威来源。
- 任何新增、编辑、删除题目的操作都必须通过 DataStore 写回源 Markdown，再重新解析题库。
- UI 层不得只修改内存状态来模拟内容变更。

### 8.2 插件数据结构

插件数据需要保存：

- decks：题库缓存，按源文件路径索引。
- settings：插件设置。
- studyHistory：学习历史。
- lastSync：最近同步时间。

题库数据需要保存：

- 题库 ID、名称、源文件路径、标签。
- 卡片列表。
- 学习次数。
- 最后学习时间。

卡片数据需要保存：

- 卡片 ID。
- 正面、背面、解释。
- FSRS 卡片状态。
- 源文件路径。
- 源文件中的顺序索引。

### 8.3 兼容迁移

- 系统必须兼容旧设置字段 `flashcardTag` 到 `flashcardTags` 的迁移。
- 系统必须兼容旧卡片字段 `question`、`answer` 到 `front`、`back` 的反序列化。
- 当前 PRD 不要求自动迁移旧卡片源文件格式。

## 9. 非功能需求

### 9.1 性能

- 题库同步应尽量对每个文件只读取一次。
- 单词列表必须使用虚拟滚动，并支持变高行。
- 单词列表滚动状态更新应使用 requestAnimationFrame 节流。
- 单词列表应在容器宽度变化时重置行高缓存。
- React 组件应避免不必要的重复渲染，长列表项可使用 memo 和内容可见性优化。

### 9.2 可用性

- 核心学习流程必须支持键盘操作。
- 主要按钮必须有清晰的 title 或 aria-label。
- 删除和退出学习等高风险操作必须有确认。
- 移动端应提供适配的返回控件和不依赖 hover 的核心交互。
- 空状态、加载状态和错误通知必须清晰。

### 9.3 安全与内容渲染

- Markdown 内容必须通过 Obsidian 的 MarkdownRenderer 渲染。
- 结构标记独占行必须被编辑器拦截，避免破坏源文件解析。
- 源文件不存在、题库不存在、卡片不存在等异常必须以通知或安全返回处理。

### 9.4 兼容性

- 插件必须支持桌面端和移动端 Obsidian。
- 插件不得依赖 Node.js 全局变量在移动端运行。
- 插件构建产物必须保持 Obsidian 插件根目录加载契约。
- 设置页必须在不同 Obsidian 设置 API 行为下保持可见。

## 10. 边界与风险

### 10.1 当前边界

- 当前产品不提供云同步，依赖 Obsidian vault 和用户自己的同步方案。
- 当前产品不提供跨设备冲突解决。
- 当前产品不提供富文本编辑器，题目编辑使用 Markdown 文本域。
- 当前产品不自动转换旧卡片格式。
- 当前刷题模式不影响 FSRS 学习进度。
- 当前统计仅保留最近 20 个有记录日期。

### 10.2 主要风险

- 卡片 ID 基于源文件路径和卡片顺序索引，用户大规模重排源文件可能影响历史状态匹配。
- 用户手动在源文件中写入非法结构标记可能导致解析结果与预期不一致。
- 删除题目后后续卡片 ID 会重映射，需要持续保证学习和刷题队列同步更新。
- 移动端 WebView 环境对部分浏览器全局对象、SVG 图标和布局行为更敏感，需要持续回归。
- 设置页依赖 Obsidian API，不同版本环境下需要保留兜底渲染逻辑。

## 11. 验收标准

### 11.1 题库扫描验收

- 给定一个首个标签为配置标签且包含 `??` 和 `;;` 的 Markdown 文件，刷新后首页出现对应题库。
- 给定一个首个标签不在配置中的文件，刷新后首页不出现该题库。
- 给定一个没有合法卡片格式的文件，即使包含配置标签，也不生成题库。
- 给定包含中文标签的文件，插件能正确识别标签。

### 11.2 学习模式验收

- 用户进入题库后能看到今日新卡数和待复习数。
- 用户选择反转模式后，问题区域展示背面，答案区域展示正面。
- 用户显示答案前看不到答案和解释。
- 用户评分后卡片状态被持久化，刷新视图后进度不丢失。
- 用户选择重来后，该卡会在本轮后续再次出现。
- 用户按 `6` 能回到上一题。

### 11.3 刷题模式验收

- 用户可选择指定题量开始刷题。
- 刷题抽题数量不超过题库总数。
- 正确和错误记录能反映在总结页正确率中。
- 错题列表按本轮出题方向展示。
- 点击错题练习后只进入错题队列。
- 刷题完成不会改变卡片 FSRS due 时间。

### 11.4 单词列表验收

- 列表只常驻展示正面和背面两列。
- 点击“乱序”后列表顺序改变，点击恢复后回到源文件顺序。
- 遮盖第一列或第二列后，该列内容隐藏。
- 点击被遮盖单元格可显示该项内容。
- 双击第一列或第二列可打开解释弹窗。
- 大量卡片时滚动流畅，不一次性渲染全部行。

### 11.5 内容维护验收

- 新增题目后，源 Markdown 文件末尾出现新卡片块。
- 编辑题目后，源 Markdown 中对应卡片块被替换。
- 删除题目后，源 Markdown 中对应卡片块被移除。
- 新增或编辑时，正面和背面为空会阻止保存。
- 新增或编辑时，内容包含独占行 `??`、`::`、`;;` 会阻止保存。
- 内容写回后重新刷新题库，界面展示最新内容。

### 11.6 设置和统计验收

- 用户添加标签后，同步能扫描该标签对应题库。
- 用户刷新并清理标签后，不存在的配置标签会被移除。
- 用户切换英文后，主视图和控制入口文案更新为英文。
- 完成学习、刷题或有效列表浏览后，统计页出现对应记录。
- 统计页只展示最近 20 个有记录日期的数据。

## 12. 后续迭代建议

以下内容不属于当前 PRD 的实现范围，但可作为后续讨论方向：

- 源文件卡片 ID 稳定化，降低卡片重排对学习状态的影响。
- 提供旧格式到当前格式的显式迁移工具。
- 提供题库内搜索、筛选和批量编辑能力。
- 增加更细粒度的学习统计图表。
- 支持导入、导出或与其他闪卡工具互通。
- 提供用户可配置的刷题完成文案编辑界面。
